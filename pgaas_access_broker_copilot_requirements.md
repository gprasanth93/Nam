# PGaaS Access Broker - Copilot Implementation Requirements

## 1. Service summary

Build a Node.js CommonJS service named `pgaas-access-broker`.

The public DNS for the service will be:

```text
https://pgaas-creds.{env}.digital.hsbc
```

Examples:

```text
https://pgaas-creds.uat.digital.hsbc
https://pgaas-creds.prod.digital.hsbc
```

The service is the request staging and orchestration layer between `pgaas-dhp-sidecar`, `pgaas-config`, and PgMaker.

Important rule:

```text
pgaas-access-broker must never receive, store, log, or deliver database passwords.
```

PgMaker remains the only component that generates credentials and delivers them to the sidecar `replyUrl`.

---

## 2. Main responsibility

`pgaas-access-broker` receives credential requests from sidecars, performs validation and later workload/OIDC authentication, resolves database configuration using `pgaas-config`, ensures the database exists through PgMaker, and finally forwards the credential request to PgMaker.

PgMaker then sends credentials directly to the sidecar callback URL.

High-level flow:

```text
pgaas-dhp-sidecar
    -> pgaas-access-broker
    -> pgaas-config
    -> PgMaker database ensure/create
    -> PgMaker credential request
    -> PgMaker sends credentials to sidecar replyUrl
```

The sidecar does not know the database name when it sends the request. It only sends application identity hints such as `git_repo_id`, `EIMID`, optional `git_repo_name`, optional database/VM config, and `replyUrl`.

---

## 3. Key design rule: per-binding FIFO queue

The service must prevent request flooding.

If 20 service VMs/sidecars request credentials for the same repo/EIMID/database binding at the same time:

```text
1st request becomes the active request for that binding key.
Other 19 requests are staged in FIFO order.
The active request performs config resolution, DB creation/waiting, and PgMaker credential forwarding.
If the active request completes, the broker forwards the remaining staged requests to PgMaker.
If the active request fails after retries/timeouts, it is marked failed and the next staged request is promoted.
```

Use queue terminology, not leader/follower terminology.

Preferred terminology:

```text
active request
queued request
binding queue
request promotion
single active workflow per binding key
```

Do not use these terms in code or docs:

```text
leader
follower
```

---

## 4. Binding key

A binding key identifies the app/database mapping.

Compute it from stable fields:

```text
source_platform + provider + environment + git_repo_id + EIMID + database_key
```

Example:

```text
apphsbc:dhp-pgmaker:uat:123456:EIM1234567:default
```

Rules:

- `git_repo_id` is stable and required.
- `EIMID` is required.
- `git_repo_name` is optional and is used only for human-readable naming/logging.
- `database_key` defaults to `default`.
- Do not use `git_repo_name` as a unique key because repo names can be changed.

---

## 5. Database name strategy

`pgaas-config` owns database name allocation and uniqueness.

The access broker should not generate final DB names by itself.

Recommended DB name generated by `pgaas-config`:

```text
pg_<repo_slug>_<env>_<hash8>
```

Example:

```text
pg_payment_service_uat_a8f31c2d
```

Where:

```text
repo_slug = normalized repo name at first creation
hash8 = hash(source_platform + git_repo_id + EIMID + environment + database_key)
```

If repo name changes later, do not rename the database. Store both:

```text
repo_name_at_creation
current_git_repo_name
```

## 9. Queue processing model

### 9.1 Request insertion logic

When a request comes in:

```text
1. Validate payload.
2. Later: validate Authorization Bearer token.
3. Build binding_key.
4. Insert request row with status RECEIVED.
5. Try to make it active if no active request exists for this binding_key.
6. If active slot is available, mark request ACTIVE and enqueue processing.
7. If another request is active, mark request QUEUED.
8. Return immediately to sidecar.
```

### 9.2 Active request processing

Only one request per `binding_key` can be active at a time.

Active request does:

```text
1. Resolve or create binding in pgaas-config.
2. If DB does not exist, send PgMaker ensure/create request.
3. Poll PgMaker until database is ready.
4. Forward this request to PgMaker credentials/request.
5. Mark request FORWARDED_TO_PGMAKER.
6. Process queued requests for the same binding_key.
```

### 9.3 Queued request behavior

Queued requests should not call `pgaas-config` or PgMaker while another request is actively creating/resolving the database.

When the active request completes:

```text
1. Fetch all queued requests for the same binding_key in created_at order.
2. If DB is now ready, forward each queued request to PgMaker credentials/request.
3. Mark each as FORWARDED_TO_PGMAKER.
4. Do not request credentials from PgMaker as a batch unless PgMaker supports batch delivery.
5. PgMaker sends credentials to each request's replyUrl.
```

### 9.4 Failure behavior

If the active request fails after configured retries/timeouts:

```text
1. Mark active request FAILED.
2. Do not leave the queue blocked.
3. Promote the next queued request for the same binding_key.
4. Start processing the promoted request.
```

This prevents the remaining 19 requests from staying on hold forever.

## 10. Retry and timeout policy

Use retry with exponential backoff and jitter for dependency calls.

### 10.1 Defaults

```text
HTTP request timeout: 30 seconds
pgaas-config resolve retries: 3
PgMaker ensure retries: 3
PgMaker operation poll interval: 10 seconds
PgMaker operation max wait: 15 minutes
PgMaker credential request retries: 5
Base retry delay: 1 second
Max retry delay: 30 seconds
Jitter: enabled
```

### 10.2 Retryable errors

Retry on:

```text
HTTP 408
HTTP 429
HTTP 500
HTTP 502
HTTP 503
HTTP 504
network timeout
connection reset
connection refused
temporary DNS failure
```

Do not retry on:

```text
HTTP 400
HTTP 401
HTTP 403
HTTP 404, unless endpoint-specific logic says resource is still being created
HTTP 409 for non-idempotent conflicting config changes
validation failures
```

### 10.3 Timeout handling

If database creation exceeds max wait:

```text
1. Mark active request FAILED with reason DB_CREATE_TIMEOUT.
2. Promote next queued request.
3. The next request may find that DB creation eventually completed and can move forward quickly.
```

### 10.4 Dependency failure handling

If `pgaas-config` is down:

```text
Retry active request.
If retries exhausted, mark request FAILED and promote next queued request.
```

If PgMaker is down:

```text
Retry active request.
If retries exhausted, mark request FAILED and promote next queued request.
```

If PgMaker credential request was accepted but the sidecar did not receive credentials, the sidecar will retry. The broker should treat that as a new request and stage/process it based on the same binding key.

### 11.2 Unique active request index

This guarantees only one active request per binding key.

```sql
CREATE UNIQUE INDEX uniq_active_request_per_binding
ON pgaas_access_requests (binding_key)
WHERE queue_status = 'active'
  AND status IN (
    'RECEIVED',
    'AUTH_CHECKED',
    'CONFIG_RESOLVING',
    'CONFIG_RESOLVED',
    'DB_ENSURE_REQUESTED',
    'DB_CREATING',
    'DB_READY',
    'FORWARDING_TO_PGMAKER',
    'RETRYING'
  );
```

### 11.3 Idempotency index

Avoid duplicate rows when the same sidecar retries the same request.

```sql
CREATE UNIQUE INDEX uniq_sidecar_request_id
ON pgaas_access_requests (sidecar_id, sidecar_request_id)
WHERE sidecar_request_id IS NOT NULL;
```

## 15. Important implementation details

### 15.1 Express app

Use JSON body parsing:

```js

```

Do not log full request body because it may contain future tokens or sensitive configuration.

Log only:

```text

binding_key
source_platform
provider
environment
git_repo_id
EIMID
sidecar_id
status
```

### 15.2 Use fetch wrapper

Create `fetchWithRetry.js`.

It must support:

```text
timeout using AbortController
retryable status codes
exponential backoff
jitter
safe JSON parsing
correlation headers
```

Headers to pass:

```text
x-request-id
x-service-name: pgaas-access-broker
```

### 15.3 Worker model

Run an internal worker loop in the same process for MVP.

Later this can move to a separate worker deployment.

Worker loop:

```text
Every WORKER_POLL_INTERVAL_MS:
  1. Find active requests that are unlocked or lock expired.
  2. Lock one request using SELECT ... FOR UPDATE SKIP LOCKED.
  3. Process it.
  4. After completion or failure, promote next queued request for same binding_key.
```

### 15.4 Locking

Use database locks, not only in-memory locks.

This allows multiple broker instances to run safely.

Use fields:

```text
locked_by
locked_until
```

When worker claims a request:

```sql
UPDATE pgaas_access_requests
SET locked_by = $worker_id,
    locked_until = now() + interval '120 seconds',
    updated_at = now()
WHERE request_id = $request_id
  AND (locked_until IS NULL OR locked_until < now())
RETURNING *;
```

## 16. Request processing pseudo-code

### 16.1 Incoming request

```js
async function handleCredentialRequest(req, res) {
  const payload = req.body;

  validatePayload(payload);

  // Later: validateBearerToken(req.headers.authorization, payload)

  const bindingKey = buildBindingKey(payload);
  const requestId = generateRequestId();

  const existing = await findBySidecarRequestId(payload.sidecar.sidecar_id, payload.sidecar_request_id);
  if (existing) {
    return res.status(202).json(toRequestResponse(existing));
  }

  const created = await insertRequest(payload, bindingKey, requestId);

  const activated = await tryActivateRequest(created.request_id, bindingKey);

  if (activated) {
    return res.status(202).json({
      request_id: requestId,
      binding_key: bindingKey,
      queue_status: 'active',
      status: 'accepted',
      message: 'Request accepted and is active for this database binding.'
    });
  }

  const position = await getQueuePosition(bindingKey, requestId);

  return res.status(202).json({
    request_id: requestId,
    binding_key: bindingKey,
    queue_status: 'queued',
    position,
    status: 'staged',
    message: 'A request is already in progress for this binding. This request has been queued.'
  });
}
```

### 16.2 Active request processor

```js
async function processActiveRequest(request) {
  try {
    await updateStatus(request.request_id, 'CONFIG_RESOLVING');

    const binding = await pgaasConfigClient.resolveBinding(request);

    await saveBindingResult(request.request_id, binding);

    if (binding.database_status !== 'ready') {
      await updateStatus(request.request_id, 'DB_ENSURE_REQUESTED');

      const ensureResult = await pgmakerClient.ensureDatabase(request, binding);

      if (!ensureResult.ready) {
        await saveOperationId(request.request_id, ensureResult.operation_id);
        await updateStatus(request.request_id, 'DB_CREATING');
        await pgmakerClient.waitForOperation(ensureResult.operation_id);
      }
    }

    await updateStatus(request.request_id, 'DB_READY');

    await updateStatus(request.request_id, 'FORWARDING_TO_PGMAKER');

    await pgmakerClient.requestCredentials(request, binding);

    await markCompleted(request.request_id, 'FORWARDED_TO_PGMAKER');

    await forwardQueuedRequestsForBinding(request.binding_key, binding);

  } catch (err) {
    await handleActiveRequestFailure(request, err);
  } finally {
    await promoteNextQueuedRequest(request.binding_key);
  }
}
```

### 16.3 Forward queued requests after DB ready

```js
async function forwardQueuedRequestsForBinding(bindingKey, binding) {
  const queued = await getQueuedRequests(bindingKey);

  for (const request of queued) {
    try {
      await updateStatus(request.request_id, 'FORWARDING_TO_PGMAKER');
      await pgmakerClient.requestCredentials(request, binding);
      await markCompleted(request.request_id, 'FORWARDED_TO_PGMAKER');
    } catch (err) {
      await markFailed(request.request_id, err);
    }
  }
}
```

### 16.4 Promote next queued request

```js
async function promoteNextQueuedRequest(bindingKey) {
  const next = await getOldestQueuedRequest(bindingKey);
  if (!next) return;

  await activateRequest(next.request_id);
}
```

## 17. pgaas-config PR coalescing requirement

`pgaas-config` stores config in a Git repo.

When sidecars send config changes:

```text
1. Canonicalize requested_config JSON.
2. Calculate config_hash = sha256(canonical_json(requested_config)).
3. Use unique key: binding_id + config_hash + open PR status.
4. If same config_hash already has open PR, return existing PR.
5. If different config_hash already has open PR, return 409 CONFIG_CHANGE_ALREADY_PENDING.
```

This ensures 20 sidecars with the same new payload create only one pull request.

The broker should not create Git PRs directly. It only calls `pgaas-config`.

## 18. Security requirements

- Do not log passwords.
- Do not store passwords.
- Do not expect PgMaker to return passwords to broker.
- Do not trust `git_repo_name` for authorization.
- Later, validate sidecar auth token before staging request.
- Later, compare token claims with request body:
  - `git_repo_id`
  - `EIMID`
  - `source_platform`
  - `environment`
- Reject mismatched token/request claims.
- Only allow trusted `replyUrl` domains.
- Validate `replyUrl` starts with approved Cranker domain for the environment.
- Add rate limiting per `binding_key` and per `sidecar_id` later.

## 19. Acceptance criteria

### 19.1 Startup request with no config

Given sidecar sends only repo/EIMID/replyUrl:

```text
Broker stages request.
pgaas-config creates default config if absent.
Broker asks PgMaker to create DB if absent.
Broker waits until DB ready.
Broker sends credential request to PgMaker.
PgMaker delivers credentials to sidecar replyUrl.
Broker never sees password.
```

### 19.2 20 concurrent sidecars for same repo/EIMID

```text
Only one request is active for the binding key.
Other 19 are queued.
Only one pgaas-config create/resolve workflow happens initially.
Only one DB creation request is sent to PgMaker.
After DB is ready, all 20 requests are forwarded to PgMaker for credential delivery.
```

### 19.3 Active request failure

```text
Active request retries dependency calls.
If retries/timeouts are exhausted, active request is marked FAILED.
Next queued request is promoted automatically.
Queue is not blocked forever.
```

### 19.4 Existing database

```text
pgaas-config returns existing ready binding.
Broker does not call DB create.
Broker forwards request to PgMaker credentials/request.
```

### 19.5 Config change PR coalescing

```text
Multiple sidecars send identical requested_config.
pgaas-config creates only one pull request.
Broker continues to request credentials using currently active DB config.
```

## 20. Non-goals for MVP

Do not implement these in MVP unless explicitly requested:

```text
Credential generation
Credential delivery to sidecar
PostgreSQL wire protocol
Sidecar local proxy
Git PR creation inside pgaas-access-broker
Database naming inside pgaas-access-broker
Full OIDC integration
```

These belong to PgMaker, sidecar, pgaas-config, or future phases.
